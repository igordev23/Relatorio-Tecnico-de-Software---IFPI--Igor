O diagrama de classes da Figura~\ref{fig:diagrama-classes} modela as entidades implementadas no MVP do sistema de certificados, abrangendo as camadas de domínio, aplicação, infraestrutura e apresentação. A arquitetura segue o padrão \textit{Domain-Driven Design} (DDD) com separação clara entre as responsabilidades de cada camada.

No centro do modelo está a classe \code{Certificate}, que representa o certificado emitido. Seus atributos incluem \code{id}, \code{recipientName}, \code{recipientCPF}, \code{courseName}, \code{courseHours}, \code{verificationCode}, \code{validityDate}, \code{templateId}, \code{issuedAt}, \code{createdAt} e \code{updatedAt}. A classe expõe os métodos \code{isValid()}, \code{updateValidity(newDate)}, \code{update(details)} e \code{toJSON()}. O código de verificação único é modelado como uma classe separada \code{VerificationCode}, que encapsula a lógica de geração (\code{generate()}), reconstrução a partir de string (\code{fromString()}), comparação (\code{equals(other)}) e formatação (\code{toString()}), com constantes estáticas \code{CHARSET} e \code{LENGTH = 8}.

A classe \code{Template} gerencia os modelos de certificado, com atributos \code{id}, \code{name}, \code{description}, \code{layoutConfig} (\code{TemplateLayout}), \code{createdAt} e \code{updatedAt}. \code{TemplateLayout} é uma classe separada que define as propriedades visuais do template: \code{backgroundColor}, \code{primaryColor}, \code{borderColor}, \code{titleFontSize}, \code{bodyFontSize}, \code{showLogo} e \code{showBorder}. A relação entre \code{Certificate} e \code{Template} é N:1 (um template pode ser usado por vários certificados), e entre \code{Template} e \code{TemplateLayout} é 1:1 (composição).

As interfaces de repositório (\textit{repository contracts}) são definidas como \code{ICertificateRepository} e \code{ITemplateRepository}, estabelecendo os contratos de persistência com operações CRUD (\code{findById}, \code{findAll}, \code{create}, \code{update}, \code{delete}), além de \code{findByCPFAndCode} no repositório de certificados. A interface \code{IPDFService} define o contrato para geração de PDF com o método \code{generateCertificatePDF(certificate, template)}.

A implementação concreta dos repositórios é feita por \code{SupabaseCertificateRepository} e \code{SupabaseTemplateRepository}, que utilizam o Supabase como banco de dados. O serviço de PDF é implementado por \code{PDFKitService}, que gera os arquivos PDF dos certificados. As setas tracejadas com o estereótipo \guillemotleft implements\guillemotright\ indicam a realização das interfaces.

Os casos de uso (\textit{use cases}) da camada de aplicação são: \code{EmitCertificateUseCase} (emissão de certificado), \code{GeneratePDFUseCase} (geração do PDF), \code{VerifyCertificateUseCase} (verificação de autenticidade) e \code{GetCertificateUseCase} (consulta de certificado), cada um expondo um método \code{execute()} com seus respectivos parâmetros e retornos.

Na camada de apresentação (\textit{frontend}), as classes \code{Certificate} e \code{Template} refletem os mesmos atributos do domínio, porém com tipos serializáveis (strings para datas). \code{VerifyResult} encapsula o resultado da verificação com \code{isAuthentic}, \code{isValid} e \code{certificate}. \code{EnvioEmail} modela a estrutura de um registro de envio de e-mail com \code{id}, \code{certificateId}, \code{recipientEmail}, \code{recipientName}, \code{status} e \code{data}; no escopo do MVP, essa classe opera exclusivamente no \textit{frontend} com persistência simulada em \code{localStorage}, sem integração com provedor SMTP nem tabela correspondente no banco de dados.
